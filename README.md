## triadic workflow
Every glyph follows three movements:
1. **seed** ‚Äî initial dictation file (voice captured in default app)
2. **form** ‚Äî glyph linked to that file, edited into readable prose
3. **reflection** ‚Äî witness entry + backlinks

## summon
glyph.html#slug_name ‚Üí loads:
- slug_name_top.md (engaging description, stud line, scroll prompt, optional quote)
- slug_name.md (main voice, polished dictation with emojis and bullets)
- slug_name_bottom.md (reflection, source link, backlinks, optional quote)
- slug_name.jpg / slug_name_2.jpg / slug_name_3.jpg (images)
- slug_name.mp4 (video)

## delivery protocol
When a new dictation file is uploaded, the editor (AI) must return the glyph in **six code blocks**:

1. `glyph_name_top`  
   - Code block with the filename (no `.md` extension)  
   - Code block with the top section content

2. `glyph_name`  
   - Code block with the filename (no `.md` extension)  
   - Code block with the middle section content

3. `glyph_name_bottom`  
   - Code block with the filename (no `.md` extension)  
   - Code block with the bottom section content

### rules
- Always deliver filename and content in separate code blocks (total = six).  
- Filename blocks must **omit the `.md` extension** for compatibility with mobile Markdown editors.  
- Top section - 1: stud line + engaging description + scroll prompt + optional quote.  
- Middle section - 2: polished dictation voice, emojis for tone, bullet points for clarity, optional pull‚Äëquotes.  
- Bottom section - 3: reflection, source link to dictation file, backlinks to related glyphs, optional closing quote.
